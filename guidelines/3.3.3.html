3.3.3 Sugestia korekty błędu (AA)

Jeśli błąd wprowadzania danych zostanie automatycznie wykryty i znane są sugestie korekty, to sugestie te są dostarczane użytkownikowi, chyba że mogłoby to zagrozić bezpieczeństwu lub celowi treści.

Opis kryterium

To kryterium poziomu AA zapewnia, że użytkownicy otrzymują pomocne sugestie jak naprawić błędy w formularzach, gdy jest to możliwe i bezpieczne. Kluczowe jest, aby:

  • Błędy były automatycznie wykrywane przez system
  • Sugestie korekty były znane systemowi
  • Sugestie były dostarczane użytkownikowi
  • Bezpieczeństwo i cel treści były zachowane
  • Sugestie były konkretne i użyteczne

Sugestie korekty powinny pomagać użytkownikom w:

  • Zrozumieniu natury błędu
  • Znajomości konkretnych kroków naprawy
  • Poznaniu właściwego formatu danych
  • Wyborze z dostępnych opcji
  • Uniknięciu podobnych błędów w przyszłości

Kto odnosi korzyść

  • Osoby z ograniczeniami poznawczymi - otrzymują jasne instrukcje naprawy
  • Osoby niewidome - rozumieją dokładnie co trzeba zmienić
  • Osoby z trudnościami motorycznymi - zmniejszają liczbę prób korygowania
  • Osoby z trudnościami językowymi - otrzymują konkretne przykłady
  • Seniorzy - mają dodatkowe wsparcie przy poprawkach
  • Wszyscy użytkownicy - szybciej i łatwiej naprawiają błędy

Wyjątki bezpieczeństwa

Sugestie nie powinny być dostarczane gdy mogłoby to:

  • Narażać bezpieczeństwo systemu
  • Ułatwiać ataki na konta użytkowników
  • Ujawniać wrażliwe informacje
  • Podważać cel formularza (np. testy, quizy)

Definicje

Błąd wprowadzania danych
Informacja dostarczona przez użytkownika, która nie jest akceptowana przez system
Sugestia korekty
Konkretna wskazówka lub instrukcja jak naprawić wykryty błąd
Automatyczne wykrywanie
Proces identyfikacji błędów przez oprogramowanie bez interwencji operatora
Znane sugestie
Możliwości naprawy, które system może określić i przedstawić użytkownikowi
Bezpieczeństwo treści
Ochrona przed ujawnieniem wrażliwych informacji lub osłabieniem zabezpieczeń

Przykłady

❌ Nieprawidłowe implementacje

Błędy bez sugestii naprawy

<!-- Nieprawidłowe - tylko informacja o błędzie -->
<div class="error">
    Nieprawidłowy format daty
</div>

<div class="error">
    Błąd w adresie email
</div>

<div class="error">
    Niedozwolona wartość
</div>

Problem: Użytkownik wie że jest błąd, ale nie wie jak go naprawić.

Niejasne sugestie

<!-- Zbyt ogólne sugestie -->
<div class="error">
    Sprawdź format daty i spróbuj ponownie
</div>

<div class="error">
    Wprowadź poprawny email
</div>

<div class="error">
    Wybierz właściwą opcję
</div>

Problem: Sugestie są zbyt ogólne i nie pomagają w konkretnej naprawie.

Naruszenie bezpieczeństwa

<!-- Nieprawidłowe - ujawnia informacje o istniejących kontach -->
<div class="error">
    Konto o emailu jan.kowalski@example.com już istnieje. 
    Spróbuj zalogować się zamiast rejestrować.
</div>

<!-- Ujawnia informacje o regułach haseł -->
<div class="error">
    Hasło nie może zawierać: admin, password, 123456, 
    data urodzenia ani imienia użytkownika
</div>

Problem: Sugestie ujawniają wrażliwe informacje systemowe.

Rodzaje sugestii korekty

Sugestie formatów

Zastosowanie: Błędy w formatach dat, telefonów, kodów pocztowych

Przykład: "Wprowadź datę w formacie DD/MM/RRRR"

Implementacja: Konkretne przykłady z poprawnymi formatami

Sugestie wyboru

Zastosowanie: Błędy w wyborze z ograniczonego zestawu opcji

Przykład: "Wybierz jeden z: Opcja A, Opcja B, Opcja C"

Implementacja: Lista dostępnych wartości lub podobnych opcji

Sugestie naprawy

Zastosowanie: Błędy w częściowo poprawnych danych

Przykład: "Czy chodziło o 'Grudzień'?"

Implementacja: Inteligentne dopasowanie i sugerowane poprawki

Sugestie uzupełnienia

Zastosowanie: Niekompletne dane

Przykład: "Dodaj brakujące 4 cyfry"

Implementacja: Wskazanie co dokładnie należy dodać

JavaScript - dynamiczne sugestie

function showErrorWithSuggestion(fieldId, message, suggestion) {
    const field = document.getElementById(fieldId);
    const errorId = fieldId + '-error';
    
    // Utwórz komunikat błędu z sugestią
    let errorElement = document.getElementById(errorId);
    if (!errorElement) {
        errorElement = document.createElement('div');
        errorElement.id = errorId;
        errorElement.className = 'error';
        errorElement.setAttribute('role', 'alert');
        field.parentNode.appendChild(errorElement);
    }
    
    // Ustaw treść z sugestią
    errorElement.innerHTML = `
        <strong>Błąd:</strong> ${message}
        <br><strong>Sugestia:</strong> ${suggestion}
    `;
    
    field.setAttribute('aria-describedby', errorId);
    field.setAttribute('aria-invalid', 'true');
}

// Przykłady użycia
function validateEmail(email) {
    if (!email.includes('@')) {
        return {
            valid: false,
            message: 'Brak znaku @ w adresie email',
            suggestion: 'Dodaj @ i domenę, np: nazwa@example.com'
        };
    }
    if (!email.includes('.')) {
        return {
            valid: false,
            message: 'Brak domeny w adresie email',
            suggestion: 'Dodaj domenę po @, np: nazwa@example.com'
        };
    }
    return { valid: true };
}

function validateDate(dateString) {
    const dateRegex = /^\d{2}\/\d{2}\/\d{4}$/;
    if (!dateRegex.test(dateString)) {
        return {
            valid: false,
            message: 'Nieprawidłowy format daty',
            suggestion: 'Użyj formatu DD/MM/RRRR, np: 15/03/1990'
        };
    }
    return { valid: true };
}

Zagadnienia bezpieczeństwa

Kiedy NIE podawać sugestii

  • Logowanie - nie ujawniaj istnienia kont użytkowników
  • Hasła - nie podawaj szczegółów o regułach bezpieczeństwa
  • Dane finansowe - nie podawaj informacji o systemach walidacji
  • Testy i quizy - nie ujawniaj poprawnych odpowiedzi
  • Systemy captcha - nie ułatwiaj automatyzacji

Bezpieczne alternatywy

  • Używaj ogólnych komunikatów o błędach logowania
  • Podawaj wskazówki formatów bez ujawniania ograniczeń
  • Oferuj pomoc kontekstową zamiast konkretnych sugestii
  • Implementuj stopniowe ujawnianie informacji
  • Używaj zewnętrznych kanałów dla wrażliwych sugestii

Najlepsze praktyki

Tworzenie sugestii

  • Bądź konkretny i precyzyjny
  • Podawaj przykłady poprawnych formatów
  • Używaj prostego, zrozumiałego języka
  • Unikaj żargonu technicznego
  • Oferuj wielopoziomowe wskazówki

Prezentacja sugestii

  • Umieszczaj sugestie blisko błędnych pól
  • Używaj kontrastowych kolorów dla uwagi
  • Strukturyzuj sugestie logicznie
  • Wyróżniaj najważniejsze wskazówki
  • Zapewniaj dostępność dla technologii asystujących

Inteligentne sugestie

  • Analizuj wzorce błędów użytkowników
  • Oferuj automatyczne poprawki gdzie to możliwe
  • Implementuj funkcje "Czy chodziło o...?"
  • Udostępniaj listy dozwolonych wartości
  • Ucz się z zachowań użytkowników

Testowanie

Metody testowania

  • Test użyteczności sugestii - czy użytkownicy potrafią naprawić błędy
  • Test z czytnikami ekranu - czy sugestie są właściwie odczytywane
  • Test bezpieczeństwa - czy sugestie nie ujawniają wrażliwych danych
  • Test z różnymi typami błędów - czy wszystkie błędy mają sugestie
  • Test z użytkownikami docelowymi - osoby z różnymi niepełnosprawnościami

Lista kontrolna

  • Czy wszystkie automatycznie wykryte błędy mają sugestie?
  • Czy sugestie są konkretne i użyteczne?
  • Czy sugestie nie naruszają bezpieczeństwa?
  • Czy sugestie są dostępne dla technologii asystujących?
  • Czy użytkownicy potrafią zastosować sugestie?
  • Czy sugestie są umieszczone blisko błędów?
  • Czy język sugestii jest jasny i zrozumiały?

Narzędzia pomocne w testowaniu

  • Testy użyteczności - obserwacja rzeczywistych użytkowników
  • NVDA/JAWS - testowanie dostępności sugestii
  • Narzędzia security testing - sprawdzanie nie ujawniania danych
  • Analytics formularzy - analiza wzorców błędów
  • A/B testing - porównywanie skuteczności różnych sugestii
